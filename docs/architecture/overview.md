# AgentForge 架构概览

## 1. 系统概述

AgentForge是一个AI驱动的Fiverr运营自动化智能助理系统，采用六层架构设计，实现松耦合、高内聚的系统结构。核心技术栈为GLM-5（核心AI模型）和N8N（工作流引擎）。

**项目定位**：专为医疗器械行业资深专业人士设计的端到端AI运营助手系统，帮助用户实现Fiverr自由职业业务的全面自动化。

**核心价值**：将重复性、可流程化的任务交给AI和自动化工具处理，让用户专注于需要专业判断和创造力的高价值工作。

## 3. 各层职责与边界

### 3.1 基础设施层 (Infrastructure Layer)

**职责**：
- 提供底层计算资源和运行环境
- Docker容器编排与管理
- 本地文件系统访问
- 网络配置与安全
- 资源监控与告警

**边界**：
- 不包含业务逻辑
- 不直接暴露给上层应用
- 通过标准化接口提供服务

**关键组件**：

| 组件 | 用途 | 技术选型 |
|------|------|----------|
| Docker Desktop | 容器运行时 | Docker 24+ |
| 本地存储 | 数据持久化 | SSD 256GB+ |
| 网络 | 容器网络 | Bridge模式 |

### 3.2 数据存储层 (Data Storage Layer)

**职责**：
- 数据持久化存储
- 缓存管理
- 向量检索与语义搜索
- 知识库管理

**边界**：
- 仅提供数据访问接口
- 不包含业务逻辑
- 数据一致性由上层保证

**关键组件**：

| 组件 | 用途 | 数据类型 |
|------|------|----------|
| PostgreSQL | 结构化业务数据 | 订单、客户、内容 |
| Redis | 缓存与会话 | 会话、临时数据、限流计数 |
| Qdrant | 向量检索 | 文档嵌入、语义搜索 |
| Obsidian | 本地知识库 | Markdown文档 |
| Notion | 云端协作 | 结构化页面、数据库 |

### 3.3 AI能力层 (AI Capabilities Layer)

**职责**：
- GLM-5核心模型集成与调用
- 多模型智能路由
- Prompt模板管理
- 记忆系统与自进化
- N8N工作流执行

**边界**：
- 提供AI能力接口
- 不直接处理业务逻辑
- 模型选择由路由策略决定

**关键组件**：

| 组件 | 用途 | 特性 |
|------|------|------|
| AgentForge Core | 核心调度 | 意图识别、任务路由、记忆管理 |
| GLM-5 Integration | AI能力 | 多语言、内容创作、客户沟通 |
| N8N Engine | 工作流 | 自动化流程、事件驱动、定时任务 |

### 3.4 业务逻辑层 (Business Logic Layer)

**职责**：
- 实现核心业务功能
- 业务规则处理
- 数据转换与处理
- 事务管理

**边界**：
- 不直接访问外部API
- 通过集成接口层与外部交互
- 业务逻辑封装在模块内

**关键模块**：

| 模块 | 职责 | 主要功能 |
|------|------|----------|
| Fiverr运营引擎 | 订单管理 | 订单监控、报价生成、交付管理 |
| 社交媒体引擎 | 内容营销 | 内容生成、多平台发布、效果分析 |
| 知识管理引擎 | 知识沉淀 | 文档管理、同步、检索 |
| 客户沟通引擎 | 客户服务 | 消息处理、回复生成、翻译 |
| 项目管理引擎 | 项目跟踪 | 任务管理、进度跟踪、报告生成 |

### 3.5 集成接口层 (Integration Layer)

**职责**：
- 外部API封装与适配
- 认证与授权管理
- 速率限制处理
- 错误处理与重试

**边界**：
- 仅负责API调用
- 不包含业务逻辑
- 统一的错误处理

**关键接口**：

| 接口 | 用途 | 认证方式 |
|------|------|----------|
| Fiverr API | 订单管理 | API Key |
| YouTube API | 视频发布 | OAuth 2.0 |
| Twitter API | 推文发布 | OAuth 2.0 |
| GitHub API | 代码管理 | Personal Token |
| Notion API | 文档同步 | Integration Token |

### 3.6 用户交互层 (User Interface Layer)

**职责**：
- 用户界面展示
- 用户输入处理
- 结果呈现
- 通知推送

**边界**：
- 不包含业务逻辑
- 通过API与后端交互
- 负责UI/UX体验

**关键组件**：

| 组件 | 用途 | 技术选型 |
|------|------|----------|
| 桌面客户端 | 主要交互界面 | Tauri + React |
| Web界面 | 备用访问方式 | React + Tailwind |
| CLI | 命令行操作 | Python Click |
| 通知系统 | 消息推送 | Telegram + 桌面通知 |

```

    │                                                                        │
    │ 调用业务API                                                            │
    ▼                                                                        │
业务逻辑层 ◄─────────────────────────────────────────────────────────────────┤
    │                                                                        │
    │ 调用AI能力、访问数据                                                   │
    ▼                                                                        │
AI能力层 ◄───────────────────────────────────────────────────────────────────┤
    │                                                                        │
    │ 调用外部API、访问数据                                                  │
    ▼                                                                        │
集成接口层 ◄──────────────────────────────────────────────────────────────────┤
    │                                                                        │
    │ 数据读写                                                               │
    ▼                                                                        │
数据存储层 ◄──────────────────────────────────────────────────────────────────┤
    │                                                                        │
    │ 资源访问                                                               │
    ▼                                                                        │

```

**依赖原则**：
1. 上层可依赖下层，下层不可依赖上层
2. 同层模块间通过接口通信，避免直接依赖
3. 跨层调用必须通过标准化接口

## 6. 技术选型说明

### 6.1 技术选型总览

| 层级 | 核心技术 | 备选方案 |
|------|----------|----------|
| 基础设施层 | Docker | Podman |
| 数据存储层 | PostgreSQL + Redis + Qdrant | SQLite + Memcached |
| AI能力层 | GLM-5 + N8N | LangChain + Airflow |
| 业务逻辑层 | Python + FastAPI | Node.js + Express |
| 集成接口层 | HTTP Client + OAuth | SDK封装 |
| 用户交互层 | Tauri + React | Electron + Vue |

### 6.2 百度千帆模型选型

| 模型 | 定位 | 特点 | 适用场景 |
|------|------|------|----------|
| GLM-5 | 核心模型 | 多语言优化、内容创作 | 内容生成、客户沟通 |
| Kimi-K2.5 | 辅助模型 | 128K超长上下文 | 长文档处理、复杂分析 |
| DeepSeek-V3.2 | 辅助模型 | 代码能力强 | 代码生成、技术文档 |
| MiniMax-M2.5 | 辅助模型 | 低延迟响应 | 快速响应、实时交互 |

### 6.3 选型依据

**OpenAkita框架优势**：
- 自进化能力：每日凌晨自动执行记忆巩固和自我检查
- 多LLM动态路由：支持9家以上大模型服务商的动态热切换
- Agent Skills和MCP开放标准：支持从GitHub一键安装技能插件

**N8N工作流引擎优势**：
- 支持200多种应用程序的原生集成
- 可视化工作流编辑能力
- 开源自托管，数据安全可控

## 7. 架构设计原则

### 7.1 松耦合高内聚
- 各层职责单一，边界清晰
- 模块间通过接口通信
- 单独模块可独立升级

### 7.2 本地优先云端协同
- 敏感数据本地存储
- 核心功能离线可用
- 云端用于协作与备份

### 7.3 AI增强人工审核
- AI生成内容必须人工审核
- 关键决策需用户确认
- 保持专业形象

### 7.4 渐进式演进
- 从小规模起步
- 根据反馈迭代
- 避免过度设计

### 7.5 安全优先
- 数据加密存储
- 最小权限原则
- 完整审计日志
