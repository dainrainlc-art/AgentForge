# AgentForge 架构决策记录 (ADR)

## ADR-001: 选择GLM-5作为核心AI模型

### 状态
已采纳

### 背景
AgentForge项目需要选择一个核心AI模型来支持Fiverr运营自动化、社交媒体营销、客户沟通等核心业务场景。可选的模型包括GLM-5、Kimi-K2.5、DeepSeek-V3.2、MiniMax-M2.5，均通过百度千帆Coding Plan Pro提供。

### 决策
选择GLM-5作为核心AI模型，其他模型作为辅助。

### 理由
1. **多语言处理能力强** - GLM-5在英文和中文双语能力上表现优异，适合国际化Fiverr业务
2. **内容创作能力优秀** - 社交媒体文案生成质量高，创意性强
3. **客户沟通场景适配** - 专业且友好的沟通风格，理解客户需求准确
4. **响应效率与质量平衡** - 适合日常运营自动化场景，配额消耗合理
5. **技术生态支持** - 百度千帆Coding Plan Pro稳定支持，API可靠

### 后果
- 核心业务流程（客户沟通、内容创作、多语言处理）将默认使用GLM-5
- 长文档处理场景使用Kimi-K2.5（128K上下文）
- 代码生成场景使用DeepSeek-V3.2
- 需要实现模型路由策略和故障转移机制

## ADR-002: 选择N8N作为工作流引擎

### 状态
已采纳

### 背景
AgentForge项目需要一个工作流引擎来处理自动化任务、事件驱动流程和定时任务。需要与GLM-5协同工作，实现AI生成与流程执行的分离。

### 决策
选择N8N作为核心工作流引擎。

### 理由
1. **开源免费** - 自托管无额外费用，适合个人项目
2. **可视化编辑** - 低代码工作流设计，便于维护
3. **丰富的集成** - 200+原生节点，支持主流平台
4. **自托管部署** - 数据安全，完全控制
5. **与GLM-5协同** - 可通过HTTP Request节点调用GLM-5 API

### 后果
- 需要部署N8N容器
- 工作流定义存储在N8N中
- 需要设计GLM-5与N8N的协同接口
- 复杂业务逻辑通过工作流实现

## ADR-003: 采用六层架构设计

### 状态
已采纳

### 背景
AgentForge是一个复杂的AI驱动系统，涉及多个子系统和技术栈。需要清晰的架构分层来确保系统的可维护性和可扩展性。

### 决策
采用六层架构设计：基础设施层、数据存储层、AI能力层、业务逻辑层、集成接口层、用户交互层。

### 理由
1. **职责分离** - 每层职责明确，降低耦合
2. **独立演进** - 各层可独立升级和替换
3. **测试友好** - 各层可独立测试
4. **团队协作** - 清晰的边界便于分工
5. **扩展性强** - 新功能可按层添加

### 后果
- 需要定义清晰的层间接口
- 跨层调用需要遵循依赖规则
- 初期架构设计工作量较大
- 长期维护成本降低

## ADR-004: 本地Docker部署方案

### 状态
已采纳

### 背景
AgentForge的目标部署环境是Microsoft Surface Pro 7+本地设备。需要选择合适的部署方案来简化部署和管理。

### 决策
采用Docker Compose进行本地容器化部署。

### 理由
1. **环境一致性** - 开发和生产环境一致
2. **快速部署** - 一键启动所有服务
3. **资源隔离** - 容器间资源隔离
4. **易于维护** - 统一的配置管理
5. **本地优先** - 敏感数据不出本地

### 后果
- 需要安装Docker Desktop
- 需要WSL2后端支持
- 资源开销略高于裸机部署
- 便于后续迁移到云环境

## ADR-005: PostgreSQL作为主数据库

### 状态
已采纳

### 背景
AgentForge需要存储订单、客户、内容等结构化业务数据。需要选择合适的关系型数据库。

### 决策
选择PostgreSQL作为主数据库。

### 理由
1. **开源免费** - 无许可费用
2. **功能丰富** - JSONB、全文搜索、向量扩展
3. **性能优异** - 复杂查询性能好
4. **生态成熟** - 工具链完善
5. **与Qdrant互补** - 结构化数据用PG，向量数据用Qdrant

### 后果
- 需要设计数据库Schema
- 需要定期备份
- 需要监控数据库性能
- 可使用pgvector扩展支持向量搜索

## ADR-006: Redis作为缓存层

### 状态
已采纳

### 背景
AgentForge需要缓存会话数据、API响应、LLM调用结果等，以提高响应速度和节省API配额。

### 决策
选择Redis作为缓存层。

### 理由
1. **高性能** - 内存存储，响应极快
2. **数据结构丰富** - String、Hash、List、Set等
3. **持久化支持** - AOF/RDB持久化
4. **限流支持** - 原生支持限流计数
5. **生态成熟** - 客户端库丰富

### 后果
- 需要管理内存使用
- 需要设计缓存失效策略
- 需要监控缓存命中率
- 需要定期备份

## ADR-008: AI增强人工审核模式

### 状态
已采纳

### 背景
AgentForge生成的内容（客户回复、社交媒体内容等）需要确保质量和专业性。完全自动化可能带来风险。

### 决策
采用AI增强人工审核模式：AI生成内容，人工审核后执行。

### 理由
1. **质量控制** - 确保内容专业性和准确性
2. **风险规避** - 避免AI生成不当内容
3. **品牌保护** - 维护专业形象
4. **合规要求** - 满足平台规则
5. **用户信任** - 增强用户对系统的信任

### 后果
- 需要设计审核队列和工作流
- 增加人工操作环节
- 响应时间略长于完全自动化
- 质量和安全性得到保障

## ADR-009: Tauri作为桌面客户端框架

### 状态
已采纳

### 背景
AgentForge需要一个桌面客户端作为主要交互界面。需要选择合适的技术栈。

### 决策
选择Tauri作为桌面客户端框架，前端使用React。

### 理由
1. **轻量级** - 打包体积小，内存占用低
2. **跨平台** - 支持Windows/macOS/Linux
3. **安全性** - Rust后端，安全性高
4. **现代技术栈** - 支持React/Vue等前端框架
5. **本地能力** - 可访问本地文件系统

### 后果
- 需要安装Rust环境
- 需要学习Tauri API
- 打包需要针对不同平台
- 开发体验良好

## ADR-010: 模块化技能系统设计

### 状态
已采纳

### 背景
AgentForge需要支持多种业务能力（Fiverr自动化、社交媒体、知识管理等），需要灵活的扩展机制。

### 决策
采用模块化技能系统设计，每个技能独立封装。

### 理由
1. **可扩展** - 新增技能不影响现有系统
2. **可维护** - 技能独立开发和测试
3. **可复用** - 技能可在不同场景复用
4. **可配置** - 技能可动态启用/禁用
5. **职责清晰** - 每个技能专注单一功能

### 后果
- 需要定义技能接口规范
- 需要设计技能注册机制
- 需要管理技能依赖
- 系统灵活性增强
