# AgentForge Agent协作机制

## 协作模式定义

### 1. 主导-支持模式 (Lead-Support)

**适用场景**: 任务有明确的专业领域归属，需要其他专业支持

**工作流程**:
```
用户请求 → 任务分类 → 主导Agent接收 → 主导Agent请求支持 → 支持Agent响应 → 主导Agent整合 → 输出结果
```

**角色分工**:
- **主导Agent**: 负责任务整体规划、进度控制、结果整合
- **支持Agent**: 提供专业领域建议、技术支持、审核反馈

**示例**:
| 任务类型 | 主导Agent | 支持Agent |
|----------|-----------|-----------|
| 系统架构设计 | 架构师 | AI工程师、安全工程师 |
| API开发 | 软件工程师 | 网络工程师、安全工程师 |
| UI设计 | 前端工程师 | 心理学家、人类学家 |
| LLM集成 | AI工程师 | 软件工程师、网络工程师 |

### 2. 并行协作模式 (Parallel)

**适用场景**: 任务可分解为独立子任务，各子任务无强依赖

**工作流程**:
```
用户请求 → 任务分解 → 并行分配 → 各Agent独立执行 → 结果汇总 → 整合输出
```

**并行规则**:
- 子任务间无数据依赖
- 各Agent独立工作，不阻塞
- 设置超时机制，避免等待过久
- 结果汇总时处理冲突

**示例**:
| 任务 | 并行子任务 | 执行Agent |
|------|------------|-----------|
| 项目初始化 | 文档创建、代码框架、测试配置 | 软件工程师、前端工程师、测试工程师 |
| 安全审计 | API安全检查、数据加密审查、权限验证 | 安全工程师、网络工程师、软件工程师 |

### 3. 串行协作模式 (Sequential)

**适用场景**: 任务有明确的阶段顺序，后一阶段依赖前一阶段输出

**工作流程**:
```
用户请求 → 阶段1 → 阶段2 → 阶段3 → ... → 最终输出
```

**串行规则**:
- 明确阶段边界和交付物
- 设置阶段检查点
- 上一阶段完成后触发下一阶段
- 异常时回退或暂停

**示例**:
| 任务 | 阶段顺序 | 执行Agent |
|------|----------|-----------|
| 新功能开发 | 架构设计 → API设计 → 实现 → 测试 | 架构师 → 软件工程师 → 测试工程师 |
| 内容发布 | 内容创作 → 本地化 → 审核 → 发布 | AI工程师 → 人类学家 → 心理学家 → 网络工程师 |

## 信息共享规范

### 共享记忆区 (Shared Memory)

**存储内容**:
- 项目全局信息
- 技术决策记录
- 公共配置参数
- 里程碑和进度

**访问权限**: 所有Agent可读可写

**文件位置**: `.trae/memory/shared/`

### 角色专属记忆 (Role Memory)

**存储内容**:
- Agent专业知识库
- 历史工作记录
- 个人偏好设置
- 领域特定上下文

**访问权限**: 仅对应Agent可读写，其他Agent只读

**文件位置**: `.trae/memory/agents/{agent_id}/`

### 任务上下文 (Task Context)

**存储内容**:
- 当前任务描述
- 相关历史记录
- 中间结果缓存
- 依赖关系图

**访问权限**: 任务相关Agent可访问

**传递机制**: 任务分配时自动注入

## 冲突解决机制

### 冲突类型

1. **技术决策冲突**: 不同Agent对技术方案有不同意见
2. **资源竞争冲突**: 多个Agent需要同一资源
3. **优先级冲突**: 任务优先级判断不一致
4. **输出冲突**: 并行任务的输出存在矛盾

### 解决策略

| 冲突类型 | 解决策略 | 决策者 |
|----------|----------|--------|
| 技术决策冲突 | 专业领域优先原则 | 领域Agent主导 |
| 跨领域技术冲突 | 架构师裁决 | architect |
| 资源竞争冲突 | 优先级队列 + 时间片 | 系统自动 |
| 优先级冲突 | 用户确认 | 用户 |
| 输出冲突 | 版本合并 + 人工审核 | 用户 |

### 冲突处理流程

```
冲突检测 → 冲突分类 → 策略选择 → 执行解决 → 记录日志
    ↓
如果无法自动解决 → 通知用户 → 用户决策 → 执行决策
```

## Agent间通信协议

### 消息格式

```json
{
  "message_id": "uuid",
  "timestamp": "ISO8601",
  "from_agent": "agent_id",
  "to_agent": "agent_id | broadcast",
  "message_type": "request | response | notification | error",
  "task_id": "task_uuid",
  "content": {
    "action": "action_name",
    "data": {},
    "context": {}
  },
  "priority": "high | medium | low",
  "requires_response": true
}
```

### 通信规则

1. **同步通信**: 用于需要立即响应的请求
2. **异步通信**: 用于非紧急的通知和信息共享
3. **广播通信**: 用于全局通知和状态更新
4. **点对点通信**: 用于Agent间的直接协作

### 超时与重试

- 同步请求超时: 30秒
- 异步消息确认: 5秒
- 重试次数: 3次
- 重试间隔: 指数退避
