## 背景

### 问题描述

AgentForge 项目当前实现与《AgentForge 个人助理系统架构规划方案 V1.0》存在严重偏离，主要体现在以下方面：

1. **核心框架缺失**: 规划要求采用 OpenAkita 作为核心框架，当前实现为自研 agentforge-core 框架
2. **自进化能力缺失**: 完全缺失记忆巩固、自我检查、自动复盘机制
3. **n8n 工作流引擎未集成**: 规划要求集成 n8n 实现可视化工作流编排
4. **外部 API 集成缺失**: Fiverr/LinkedIn/Notion/Obsidian 等外部 API 直连未实现

### 约束条件

**技术约束**:
- 需要兼容现有数据库设计
- 需要保持前端 UI 可用
- 需要延续百度千帆 AI 集成

**业务约束**:
- 重建期间需要保持业务连续性
- 数据迁移需要保证完整性
- 重建周期控制在 12-16 周内

**时间约束**:
- 越晚重建成本越高
- 技术债务持续累积
- 业务价值延迟实现

### 利益相关者

- 项目负责人: 最终决策者
- 开发团队: 执行重建工作
- 目标用户: 系统使用者

---

## 决策

### 决策内容

采用**渐进式重建策略**，保留可用模块，逐步重构核心架构。

**重建方案**:

| 模块类型 | 模块名称 | 处理方式 |
|----------|----------|----------|
| 保留 | 前端 UI | 继续使用 |
| 保留 | 数据库设计 | 保持兼容 |
| 保留 | AI 集成 (百度千帆) | 继续使用 |
| 重构 | API 层 | 逐步优化 |
| 重构 | 业务逻辑层 | 逐步优化 |
| 重建 | 核心框架 | 引入 OpenAkita |
| 重建 | 自进化系统 | 全新实现 |
| 重建 | 工作流引擎 | 集成 n8n |

### 决策理由

1. **业务连续性**: 现有系统可以持续使用，不影响日常工作
2. **风险可控**: 每个阶段独立验证，问题可以及时发现和修复
3. **资源优化**: 充分利用现有代码，避免浪费
4. **灵活调整**: 可以根据实际情况调整重建策略
5. **成本可控**: 渐进式投入，可以根据预算调整进度

### 实施计划

```
阶段 0: 准备阶段 (Week 1-2)
├── OpenAkita 架构研究
├── 迁移方案设计
├── 开发环境搭建
└── 团队培训

阶段 1: 核心重建 (Week 3-8)
├── OpenAkita 核心集成
├── 自进化机制实现
├── n8n 工作流引擎集成
└── Fiverr API 直连

阶段 2: 功能完善 (Week 9-12)
├── 事件驱动架构实现
├── Notion/Obsidian 集成
├── Docker 容器化
└── Agent Skills 系统

阶段 3: 优化上线 (Week 13-16)
├── 性能优化
├── 安全审计
├── 用户验收测试
└── 生产环境部署
```

## 后果

### 正面影响

- **核心价值实现**: 自进化能力使系统成为真正的"数字分身"
- **运营效率提升**: 自动化程度从 10% 提升至 90%
- **扩展能力增强**: Agent Skills 生态支持快速扩展
- **维护成本降低**: 自进化机制减少人工干预
- **技术债务清除**: 架构清晰，长期维护成本可控

### 负面影响

- **短期投入增加**: 需要额外的开发资源
- **学习曲线**: 团队需要学习 OpenAkita 框架
- **过渡期复杂**: 新旧架构并存期间维护复杂

### 风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| OpenAkita 集成困难 | 中 | 高 | 提前研究、准备备选方案 |
| 数据迁移失败 | 低 | 高 | 充分测试、保留回滚方案 |
| 进度延期 | 中 | 中 | 预留缓冲时间、敏捷调整 |
| 团队适应期 | 中 | 低 | 提前培训、文档完善 |

### 技术债务

**引入的技术债务**:
- 过渡期间需要维护两套代码
- 部分模块可能需要临时适配层

**管理策略**:
- 明确债务清单和清理计划
- 每个阶段结束时评估债务状态
- 优先清理高风险债务

---

## 替代方案

### 方案 A: 完全重建

- **描述**: 从零开始，完全按照规划架构重新构建系统
- **优点**: 
  - 架构最清晰
  - 技术债务最少
  - 完全符合规划
- **缺点**: 
  - 时间成本高 (16-20 周)
  - 风险大
  - 浪费现有代码
  - 业务中断
- **未选择原因**: 业务连续性风险高，资源浪费严重

### 方案 B: 混合重建

- **描述**: 核心模块重建，外围模块保留
- **优点**: 
  - 平衡时间和质量
  - 核心架构清晰
- **缺点**: 
  - 新旧架构并存
  - 集成复杂度高
- **未选择原因**: 集成复杂度不可控，风险较高

### 方案 C: 仅修复偏离点

- **描述**: 仅修复当前发现的偏离点，不进行系统性重建
- **优点**: 
  - 投入最小
  - 风险最低
- **缺点**: 
  - 核心问题未解决
  - 技术债务持续累积
  - 无法实现核心价值
- **未选择原因**: 无法解决核心架构问题，无法实现"数字分身"价值
